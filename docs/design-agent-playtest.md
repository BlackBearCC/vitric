# 设计稿：agent 集群试玩（vitric playtest）

状态：**全 6 阶段已落地**（2026-06-16，crate `vitric-playtest` + `vitric playtest` 子命令，637 测试绿）。dogfood 真实例子(spire/echo)硬化过三处假阳性(Position 误报 runaway、Light/初值0 误报 collapse、种子探索不重放 replies)。已知精化(留后续)：声明结局集合靠静态扫规则 emit，**脚本/LLM 发结局的游戏(如 echo)要在 `playtest.json` 声明 terminal/视图**才判得出 ending_coverage——这是 Phase 6 视图覆盖的用法，不是 bug。定位：**把"确定性 + gate 证书 + AI 控制面"这三张别人没有的牌，复利成一件别的引擎给不了的能力——给单机游戏配一个自动 QA + 配平师。** 放一群确定性 agent 把游戏自动玩穿，聚合出"哪通不了、哪不平衡、哪卡死、哪是废内容、节奏在哪崩"的报告，每局都是可重放可认证的录像。

支持**所有经典类型**（平台跳跃/解谜/卡牌构筑/roguelike/RPG/模拟经营/叙事冒险/动作射击/生存合成），靠的是一套**类型无关的接口** + 一个**可插拔的策略库** + **种子式探索**，不是为每个类型写一套。

## 边界：清地板，不判天花板（写在最前，定死预期）

agent **不"体验"游戏**——好不好玩、手感爽不爽、画面美不美、剧情动不动人，那是人的事，本系统不碰、不吹。它验的是**"体验得以成立的地基"**：像结构工程师，不判屋子美不美，只保证地板不塌、每扇门开得了、没有房间进不去、没人被逼挤一个瓶颈。

- **能逮（结构问题，会直接杀死体验）**：软锁/无解存档、死路、数值崩、一招鲜压死其他玩法（无意义选择=无聊）、难度悬崖、不可达的结局/内容、从没被用到的废道具废技能、剧情前后矛盾（LLM 档）。
- **不能判（要人）**：乐趣、手感、美、故事共鸣、一个笑话好不好笑。
- **价值**：让"人来体验"变值钱——人别再玩到第 3 小时才撞软锁、一下午白费；机械破绽提前清干净，人专心判天花板。反向还放大人玩：人试出"这段感觉不对"，因每局是确定性录像，可回放那一刻、调数值、让 swarm 在附近跑一圈，诊断是不是数值/节奏的锅。

## 一、场景视图 Scene View（中心件，整个系统的接口）

引擎为当前世界态自动吐一份**机器可读的"代理所见"**——策略档和 LLM 档共用同一份。三部分：

- **观测 observation**：当前相关状态。从世界快照投影，**剔除纯装饰**（render-only：Sprite 颜色/贴图、粒子、Bloom 等），保留玩法组件（Position/Health/Inventory/经济/flag…）。字段尽量人话标签。
- **动作 actions**：本刻可选的动作清单 = 游戏声明的输入词汇（规则的 input 触发 / schema 声明的输入），按当前合法性过滤（能量不够的牌不列）。
- **目标/终止 goal·done**：来自规则里标的胜负/结局事件（`ending-*`/win/lose）。done 时标明结局类型（通关/死亡/软锁/超时）。

**自动推 + 可选覆盖**：默认从 schema/rules 自动推一份能跑的（任何游戏开箱即试）；游戏可选在 `playtest.json`（或 schema 块）里覆盖成更顺的视图——挑相关字段、改人话标签、声明派生量（"距出口曼哈顿距离"）。覆盖是打磨，不是前提。

Scene View 是**纯投影**，不进模拟状态/哈希，不影响确定性。

## 二、两档代理

- **策略档（廉价 · 进程内 · 确定 · 主力）**：内置一个**策略库**，每个策略是消费 Scene View、产出动作的纯逻辑（PCG 播种→可复现）。不烧 LLM，跑几千局做广度覆盖、找崩。库（可扩展）：
  - `random`：合法动作里均匀随机（覆盖 + 找意外软锁）。
  - `greedy`：朝 goal 的派生量贪心（奔出口/打敌人/攒资源）。
  - `coverage`：系统性试遍每个动作至少一次（找废内容/废动作）。
  - `seed-perturb`：从种子录像出发，定点扰动（换序/走岔/改选择）——解谜/剧情/RPG 的深度主力（见第三节）。
  - `economy`：模拟经营专用，最大化/压垮经济找数值崩。
  - 类型特化策略按需加（物理采样 for 平台、出牌评估 for 卡牌）——都插同一接口。
- **LLM 档（少量 · 外呼 · 点缀）**：经现有 `VITRIC_LLM_*` 钩子，读同一份 Scene View，拟人玩 + 吐定性 note（"我看不懂该干嘛""这选项好像没意义""管理员这句和上一幕矛盾"）。**LLM 推理不确定，但它选的输入照样录进录像→可重放复现**，所以它找到的问题可定位。少量局（成本）。

## 三、种子式探索（gate 证书 = 种子）

真正的难题谜，随机/贪心解不出来（解法是特定 N 步序列）；从零解任意谜是没解的 AI 难题。所以对有谜深的类型，**A 的活是"绕着开发者给的解法找破绽"，不是替你从头解**。

- **种子来源**：gate 证书本身——它是一段"证明这局能通"的确定性录像。拿它当起点。
- **扰动**：`seed-perturb` 在种子的输入序列上做受控变异——换动作顺序、在分叉点走另一条、改一个对话选择、跳过一步、提前/延后用某道具——每个变异跑成一局，看是否软锁/不可达/坏结局。
- **无谜深的类型**（养成/数值/卡牌）：策略档可从零跑，种子非必需。

种子式探索把"通关证书"和"试玩 swarm"接成一条线：证书既证明能通，又当 swarm 的起点。

## 四、每局 = 确定性录像

每局试玩复用现有录像/重放基建：录 inputs（+ 用到的 replies）+ 结局标记（通关/软锁/超时/死亡）+ 关键 tick 的 checkpoint 哈希。**每局可重放、可认证**。找到的软锁 = 一段能稳定复现该软锁的录像，直接能交给开发者重放看。

## 五、聚合与报告

收 N 局聚合成**地板报告**（机器 JSON + 人话摘要）：

- **通关率 / 结局分布**：多少局到了哪个结局；哪些结局 0 局可达（不可达结局）。
- **软锁图**：哪些输入序列/状态进了无解死局；按触发条件聚类（"先用钥匙开抽屉→北门永锁，37 局命中"）。
- **死路 / 不可达内容**：到不了的房间/关卡/分支。
- **废内容**：在所有局里命中率极低的道具/技能/动作/分支（疑似没接上）。
- **难度 / 节奏曲线**：每段的耗时/重试/死亡分布——逮难度悬崖、挫败尖峰、平淡死区。
- **一招鲜 / 主导策略**：某动作/牌/build 胜率碾压其他（无意义选择）。
- **数值崩**：经济跑飞/崩盘、资源永远凑不齐、属性溢出。
- **LLM 定性 note**：清晰度/连续性/选择有效性，附触发局录像。

每条结论都挂**可重放录像**，不是空断言。

## 六、覆盖所有经典类型

类型无关靠**一套接口 + 策略库 + 种子**；报告维度大体通用，下表给每类的"地板靶子"与主力策略：

| 类型 | 主要地板靶子 | 主力策略 |
|---|---|---|
| 平台跳跃 | 够不到的平台、掉坑无重生软锁、像素级跳的挫败、难度悬崖 | greedy(空间)/seed-perturb |
| 解谜 | 软锁、不可达解、废机关 | seed-perturb（必需） |
| 卡牌/构筑 | 废牌、主导牌、无解开局、坏 combo | greedy/coverage |
| roguelike | 跨局难度曲线、主导 build、废物品、无解种子 | random/greedy 多局 |
| RPG | 任务软锁(杀了后面要的 NPC)、不可达任务、经济崩、废技能、等级门 | seed-perturb/economy |
| 模拟经营 | 经济崩/跑飞、主导策略、X 之后无目标、破产软锁 | economy |
| 叙事冒险 | 不可达结局、废分支、flag bug、连续性、软锁 | seed-perturb + 分支覆盖 + LLM |
| 动作/射击 | 打不过的遭遇、难度悬崖、坏判定(永不掉/扣血)、废武器 | greedy/反射策略 |
| 生存/合成 | 拿不到的资源(科技软锁)、废配方、饿死/跑飞 | coverage/economy |

新类型 = 接口不变，缺什么策略就往库里加一个，不动框架。

## 七、架构（A2：进程内 + LLM 外呼）

`vitric playtest <项目> [选项]` 子命令。

- **进程内跑 sim + 进程内策略 swarm**：无 HTTP 往返，跑几千局靠速度，直接复用 sim/录像/gate 基建。配平的价值在"几千局找崩"，要的就是这个速度。
- **LLM 档外呼**：走现有 `VITRIC_LLM_*`，少量局。
- 复用：`vitric-sim`（step/snapshot/restore/record/replay）、gate（证书=种子+认证）、Scene View 从 schema/rules 派生。
- 选项：`--sessions N`、`--strategy <名|all>`、`--seed <录像>`、`--llm N`、`--report <json路径>`。

## 八、确定性与自验

- **swarm 自身确定**：每局给定 (策略, PCG 种子, 起点) → 完全可复现；重跑 playtest 出同一份报告（LLM 档除外，但其录像照样可重放）。
- **埋雷测试游戏**：造一组**故意埋了已知缺陷**的小测试项目——一个埋软锁、一个埋不可达结局、一个埋一招鲜、一个埋废道具、一个埋经济崩——断言 playtest 报告**逐个逮出来**。清地板的能力本身被埋雷验证，不靠自述。
- 报告对同输入逐字节稳定（除 LLM 档），进 gate 可认证。

## 九、性能预算（硬要求）

跑几千局不能慢成灾：

- 一局 = 纯 sim 步进，无渲染、无 HTTP；策略决策是轻量纯逻辑。
- Scene View 投影**按需算**、只取相关组件，不每 tick 全量序列化；空场/无变化早退。
- 多局**并行**（各局独立 World，PCG 各自播种，结果与串行逐位一致——同确定性铁律）。
- 聚合是 O(局数 × 事件) 的一趟，不做平方比对。
- LLM 档天然慢，单独限流并发，不拖累策略档。

## 十、刻意不做

- 判乐趣/手感/美/故事共鸣（要人，见边界节）。
- 从零解任意难题谜（种子式绕着已知解找破绽，见第三节）。
- 替代真人试玩（A 清地板让人玩更值钱，不取代）。
- LLM 档跑规模（少量点缀；广度靠廉价策略档）。
- 真人在环的实时陪测（另立项）。

## 十一、分阶段（实现排序，全部在范围内，可慢迭代）

1. **地基**：Scene View 自动派生（观测/动作/目标）+ 进程内单局试玩跑通 + `random`/`greedy` 两策略 + 录像落地。验收：一个小游戏能被 agent 自动玩到通关/超时，出单局录像。
2. **swarm + 核心报告**：多局并行 + 聚合器 + 报告维度（通关率/软锁/死路/废内容/卡顿/一招鲜）+ `coverage` 策略。验收：埋雷测试游戏组，报告逐个逮出已知缺陷。
3. **种子式探索**：`seed-perturb` + gate 证书当种子 + 分支/结局覆盖。验收：解谜/剧情埋雷项目，逮出不可达结局 + 顺序软锁。
4. **类型策略库 + 数值**：`economy` 及类型特化策略 + 数值崩/主导策略维度。验收：模拟经营埋雷项目，逮出经济跑飞 + 主导策略。
5. **LLM 档**：经 `VITRIC_LLM_*` 接拟人玩 + 定性 note（清晰度/连续性）。验收：叙事项目，LLM 逮出剧情连续性矛盾，其选的输入可重放复现。
6. **每游戏视图覆盖 + 报告打磨**：`playtest.json` 覆盖、派生量、人话报告。

## 十二、验收（写给 gate）

- 每阶段的"埋雷项目逐个逮出"断言进测试。
- swarm 确定性：同 (策略,种子,起点) 重跑逐位一致；找到的软锁能重放复现。
- 性能：几千局并行在合理时间内完成（基准断言），与串行结果逐位一致。
- 报告是机器 JSON + 人话摘要双份；每条结论挂可重放录像。
- 全流程进 `vitric gate`：playtest 自身的认证 = 埋雷全逮 + 确定性重放一致。
