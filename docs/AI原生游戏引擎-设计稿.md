# Vitric — AI 原生游戏引擎设计稿

日期：2026-06-10
状态：头脑风暴定稿，待立项
目标：**做成 AI 游戏开源项目第一**

---

## 名字

**Vitric**（玻璃质的）。口号：**The glass-box game engine for AI agents**——别家引擎对 AI 是黑盒，Vitric 是玻璃盒，一切状态透明可见。GitHub/npm 无占用，单词短、可注册、语义直指核心卖点。

## 一句话定位

现有引擎（Unity/Godot/Unreal）是"人坐在编辑器前"为中心设计的；Vitric 以 **agent API 为中心**——引擎的一切状态对 AI 可见、可操作、可验证，让 **AI 能自主地运行游戏、检测问题、迭代修改，整个闭环不需要人插手**。

## 竞品格局与空白点（2026-06 调研）

市场分两堆，中间是空的：

- **消费级 prompt-to-game**（Rosebud、Bitmagic、Jabali、Astrocade）：闭源、浏览器原型级、面向无代码人群，做不出能上 Steam 的工程级游戏。
- **传统引擎贴 MCP 补丁**（unity-mcp 10.5k★、godot-mcp 4k★、Roblox 官方 MCP）：全是**编辑器期**操作，运行时状态对 AI 仍是黑盒——引擎的状态模型从根上就不是为 AI 设计的。

关键空白（调研确认无人做）：

1. **AI 自主闭环**没有完整实现。最接近的 Roblox Playtest Automation 能模拟输入、读 console，但封闭平台、无断言框架、每步要人确认。
2. **确定性回放作为 AI 调试原语**完全空白。rr/WinDbg TTD 在通用领域成熟，但没有任何游戏引擎把它做成 agent 的调试接口。这是 Vitric 最硬的差异化：别家 AI 只能"看一眼再猜"，Vitric 的 AI 能"精确重放到出 bug 前一帧"。
3. 概念上最近的 prior art 是 bevy_brp（ECS 运行时状态对 agent 可查可改），但只有 52★、无闭环、无人深耕——验证了方向可行且没被占。

要盯的对手：**Cocos 4**（2026-01 全开源，宣称 AI Native 第一优先级、功能以 Agent 形式交付，叙事重叠最大，背后有公司养）；**Summer Engine**（"AI game engine + Steam 导出"的需求验证者，但基于 Godot 做不深）。

## 要解决的三大痛点（来自 PetClaw 一年实战）

1. **AI 是瞎的**：改完效果看不见，截图要人肉来回倒腾。
2. **反馈链路断**：代码语法对了不等于跑得对——动画被打断、按钮点不动、显存爆炸，这些运行时问题在聊天里完全不可见。
3. **引擎状态不是文本**：场景树、节点连线、导入设置藏在编辑器二进制里，AI 只能摸到代码文件，碰不到引擎真正的状态。

## 核心主轴：AI 自主迭代闭环

这是整个引擎最重要的一件事，所有子系统都为它服务。闭环长这样：

```
AI 改动（数据/代码）
   ↓ 写入即校验（schema + 静态检查，错了立刻精确报错）
热重载（毫秒级，游戏不重启）
   ↓
AI 自己观察（截图 / 查任意对象状态 / 读结构化日志）
   ↓
AI 自己测试（注入输入、跑断言、无头 100 倍速回归）
   ↓
发现问题 → 录像回放精确复现 → 修 → 回到第一步
```

人只在两个地方出现：提需求，和对着运行窗口说"这里不对劲"。

闭环的四根柱子：

### 柱子 1：AI 控制面（引擎内建调试端口）

引擎进程起来就带一个本地调试端口（类似浏览器的 CDP），任何 agent 通过它能：

- **看**：截当前帧、截指定相机/区域，拿到的就是玩家看到的真实像素。
- **查**：查询任意实体、组件、规则的当前值；订阅状态变化。
- **动**：注入键鼠/手柄输入、直接调用游戏内函数、改任意组件值。
- **控时间**：暂停、单步（逐帧）、加速（10x/100x）、跳到指定时刻。
- **测**：注册断言钩子（"血量永远不该是负数"），违反时引擎主动上报。

官方配 MCP server，Claude Code / 任何 agent 开箱即用。

### 柱子 2：确定性 + 录像回放

- 模拟逻辑完全确定性：固定随机种子 + 录制全部输入 = 任何一局游戏可以逐帧精确重放。
- bug 复现从"猜+碰运气"变成"拿到录像，倒带，单步看是哪一帧坏的"。
- 回归测试白送：录一局正常游戏，每次改完代码重放，结果不一致就报警。

### 柱子 3：测试即一等公民

- 无头模式跑模拟（不渲染，百倍速），AI 写"从出生点能走到 boss 房""死亡后金币归零"这种游戏级断言当测试。
- 引擎自带"陪玩 bot"接口：AI 可以让一个脚本化/LLM 驱动的玩家把游戏从头打到尾，产出过程录像和问题清单。

### 柱子 4：一切报错为 LLM 设计

- 每条错误是结构化的：哪个实体、哪条规则、当时的相关状态快照、以及修复建议。
- 性能也是"报错"的一部分：显存、draw call、加载时间有预算，超了直接告诉 AI"哪个资源吃了多少、怎么压"。（PetClaw 8.5G 显存事故在这里只是一条编译警告。）

## 架构分层

### 1. 声明式数据层（引擎的心脏）

- 场景、实体、组件、规则、素材槽、关卡、UI、动画状态机——全部是**强 schema 的声明式数据**（基于 JSON 方言，不发明新语法）。
- **写入即校验**：引用不存在的资源、类型不对、必填缺失，提交瞬间报精确错误。
- **状态可往返**：运行中的世界状态能序列化回同一格式——AI 用什么格式写游戏，就用什么格式读正在跑的游戏。存档=快照，白送。
- 格式天生 git 友好：分块存储、diff 可读，为多 agent 并行开发铺路。
- 为什么不发明新语法：LLM 写得最好的是训练数据最多的语言，新语法零语料反而写得差。schema 是本体，语法是皮，将来真需要再加。

### 2. 逻辑层：ECS 地基 + 规则正门

- **规则是正门**：80% 的玩法是"当 X 发生时做 Y"的声明式规则直译（"碰到金币→加分"），AI 翻译需求几乎零损耗。
- **ECS 是地基**：复杂逻辑写成 TypeScript 系统函数，**必须声明自己读哪些组件、写哪些组件**——引擎因此全知全见：能静态查出两段逻辑打架、能单独测试任何系统、AI 能推断改动影响面。
- 状态永远躺在组件数据里，没有藏在对象内部的私有状态。
- 排除传统 OOP 挂脚本模式（状态互相乱摸，正是痛点 3 的根源）。
- 不写隐式 fallback：失败显式暴露，规则级联有结构兜底。

### 3. 使用模式：对话为主 + 检查器为辅

- 没有传统编辑器。人面对的是**运行窗口**：点选任何对象看属性、拖拽微调位置，改动**写回数据层**——人做的微调 AI 也看得见，双向同步。
- 人管"指着屏幕说哪里不对劲"，AI 管"怎么改"。

### 4. 素材管线：槽位 + 校验，生成靠插件

- 引擎本分：**素材槽位**——规格、尺寸、帧数、风格约束写在数据层，进来的素材严格校验（AI 始终知道缺什么、规格是什么、放进去对不对）。
- 生成是插件生态：liblib、ComfyUI、任何模型都能接。生成模型一年换三茬，不烧死在引擎里。
- prompt 和参数留档，素材可随时重生成（素材=可重新编译的源码）。
- 工程级风格锚点（调色板、参考图）防整个游戏串味。
- 导入管线强制最佳实践：自动图集、自动纹理压缩——显存膨胀类事故在导入时就拦死。

### 5. 关卡

- 关卡=分块的声明式数据，AI 直接写，人用检查器微调。
- 最小空间工具：笔刷刷地块、框选填充，结果写回数据层。
- **配方生成是招牌特性**："洞穴风格、三条岔路、难度递增"一句话生成关卡——按本架构这是规则+函数产出数据，但要打磨成开箱即用的传播级体验。

### 6. 常规子系统（每个都按"AI 看得见"重做）

- **UI**：布局声明式；可交互性静态检查——"这个按钮没接任何事件""这个 overlay 不在命中列表里"直接报警，而不是运行起来点不动。按钮反馈等基础体验是引擎默认行为。
- **动画**：动画状态机是声明式数据；"状态动画被 idle 打断"这类问题在引擎层就不可能发生，不靠使用者记咒语。
- **音频**：运行时混音；音效音乐走素材槽位+生成插件。
- **物理/碰撞、相机、粒子**：常规实现。
- **Shader**：AI 写不好的领域，配截图对比测试兜底。
- **打包发行**：一条命令出 Steam/itch 包，agent 能自己发版。

### 7. 运行时 LLM 模块（附加能力，官方模块）

- NPC 对话、动态剧情作为引擎官方模块：prompt 槽位、缓存、成本控制内建。
- 这是"AI 是内容/AI 是规则"两个方向的承载点，但优先级低于开发闭环。

### 8. 技术底座

- **核心运行时：Rust + wgpu**——性能、跨平台、Steam 原生发行。
- **逻辑层：TypeScript**，嵌入 JS 运行时执行（候选 deno_core / QuickJS，待技术验证），毫秒级热重载 + 代码改动时的状态迁移。
- 首发目标 2D 桌面游戏（Steam 可发行），架构不排除未来 3D。
- 顺带的卖点：游戏逻辑跑在沙箱里，玩家下载 AI 写的游戏不用担心代码越权。

## 开源打法

- **文档为 AI 优化**：llms.txt、一页纸 schema 规范（每次塞进上下文都不亏）、错误码全集带修法。
- **官方游戏模板** + 官方 MCP server + agent 协作规范（游戏项目的 CLAUDE.md 标准），让"clone 下来 5 分钟让 Claude 给你做个游戏"成为第一印象。
- **Dogfood 立信**：用引擎重写一个真游戏（候选：PetClaw 类桌宠或一个小型 Steam 游戏），全程 AI 开发，过程公开。
- 多 agent 协作是社区级卖点：数据分块 + git 友好 → 几个 agent 并行开发一个游戏不打架。

## 已定决策记录

| 决策点 | 结论 |
|---|---|
| AI 的角色 | AI 是开发者（主），AI 是内容/规则（附加模块） |
| 受众 | 开源社区，目标做到 AI 游戏开源项目第一 |
| 从零 vs 站肩膀 | 从零写（架构和数据模型全新设计），渲染/窗口用 wgpu 等底层库 |
| 平台 | 2D 桌面原生（Steam 可发行）+ 浏览器级 AI 可见性（内建控制面） |
| 新语言 | 不发明新语法；"AI 易用语言"=强 schema 声明式数据层 + TypeScript 逻辑层 |
| 逻辑表达 | ECS 地基 + 声明式规则正门的杂交；排除纯规则（会长成烂语言）和 OOP 脚本 |
| 人机分工 | 对话为主 + 运行窗口检查器为辅，无传统编辑器 |
| 素材 | 槽位+校验内建，生成靠插件生态 |
| 关卡 | 分块数据 + 最小笔刷 + 配方生成（招牌特性） |
| 最高优先级 | AI 自主运行→检测→迭代闭环 |

## 未决问题

1. **项目名字**（开源项目门面，要好记、域名/GitHub org 可用）。
2. JS 运行时选型（deno_core vs QuickJS vs 其他）：需要 spike 验证嵌入成本和热重载。
3. 确定性模拟与浮点跨平台一致性：技术上最难啃的骨头之一，需要早期 spike。
4. 里程碑切分：虽然目标是大项目，开发仍需顺序——建议第一里程碑就是"闭环跑通"（最小运行时 + 控制面 + 热重载 + 一个 AI 全自主做出来的小游戏 demo），这是对外的第一声。
5. 多人联机：暂不在蓝图内，确定性架构天然利好帧同步，留到后期。
