## Step 3: Add forecast-update system to hud.js

In `games/frontier/scripts/hud.js`, append the `forecast-update` system:

```javascript
// 7-day weather forecast. Updates once per day (not every tick) to avoid flicker.
// Uses ctx.random_stream("forecast") for deterministic generation — the forecast for
// day N is the same regardless of when it's generated (replay-stable via E3 substream).
//
// The forecast is a PREDICTION based on seasonal weather weights — it doesn't match the
// actual weather (which is generated by weather-tick using a separate substream). The
// forecast gives the player a rough outlook for planning.
//
// SEASONAL_WEIGHTS is duplicated from flare.js (the weather-tick system). Extracting to
// a shared module is out of scope — the weights are stable and the duplication is noted.

const FORECAST_DAYS = 7;

const FORECAST_SEASONAL_WEIGHTS = {
  spring: { clear: 50, cloudy: 30, rain: 15, storm: 5, flare: 0 },
  summer: { clear: 40, cloudy: 20, rain: 10, storm: 5, flare: 25 },
  autumn: { clear: 45, cloudy: 30, rain: 20, storm: 5, flare: 0 },
  winter: { clear: 30, cloudy: 40, rain: 0, storm: 30, flare: 0 }
};

const FORECAST_LABELS = {
  clear: "晴", cloudy: "阴", rain: "雨", storm: "暴风", flare: "耀斑"
};

function forecastPick(stream, weights) {
  const total = Object.values(weights).reduce((a, b) => a + b, 0);
  const r = stream.next() * total;
  let acc = 0;
  for (const [k, w] of Object.entries(weights)) {
    acc += w;
    if (r < acc) return k;
  }
  return Object.keys(weights)[0];
}

vitric.system("forecast-update", { query: ["Clock", "Season", "Forecast"], writes: ["Forecast"] }, (entities, ctx) => {
  for (const e of entities) {
    // Only regenerate when the day changes — not every tick.
    if (e.Clock.day === e.Forecast.last_day) continue;

    const stream = ctx.random_stream("forecast");
    const season = e.Season.current;
    const weights = FORECAST_SEASONAL_WEIGHTS[season] || FORECAST_SEASONAL_WEIGHTS.spring;

    // Generate 7 weather labels for the next 7 days.
    const labels = [];
    for (let i = 0; i < FORECAST_DAYS; i++) {
      const w = forecastPick(stream, weights);
      labels.push(FORECAST_LABELS[w] || w);
    }

    e.Forecast.days = "预报: " + labels.join(" ");
    e.Forecast.last_day = e.Clock.day;
  }
});
```

**Key points:**
- The system queries `["Clock", "Season", "Forecast"]` — all three on the `colony` entity.
- The `if (e.Clock.day === e.Forecast.last_day) continue` guard ensures the forecast only regenerates on day-change, not every tick. This prevents flicker and limits substream consumption to 7 draws per day.
- `ctx.random_stream("forecast")` is a SEPARATE substream from `ctx.random_stream("weather")` (used by `weather-tick`). The two don't interfere — forecast predictions and actual weather are independent.
- `FORECAST_SEASONAL_WEIGHTS` duplicates the weights from `flare.js`. This is a noted DRY violation — extracting to a shared module is out of scope (the weights are stable game-design constants).
- The forecast is a PREDICTION, not a guarantee. Actual weather (from `weather-tick`) will differ because it uses a different substream and updates on its own timer.
